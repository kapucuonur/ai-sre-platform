How it works
Three layers of autonomous healing
🧠

Strategy registry

Remembers every fix that ever worked. Same error on any container? Resolved instantly with zero API cost.

⚡

Local LLM first

Simple incidents routed to on-device Ollama. No network call, no latency, no cost — decision in under 2 seconds.

☁️

Cloud escalation

Complex incidents route to Gemini Flash or Claude Sonnet based on severity score. Humans only see true emergencies.

📉

Weight decay

Old fixes automatically age out. Stale solutions re-escalate to LLMs so the system never gets stuck in the past.

FAQ
Common questions
Does it need cloud access to work? +
No. The core healing loop runs entirely on your server using local Ollama. Cloud LLMs (Gemini, Claude) are only called for complex incidents you configure a budget for. If you set budget to zero, it never touches the internet.
What if the daemon makes a wrong fix? +
Every action is logged to an audit trail (heal_log.jsonl). Failed commands are automatically blacklisted in the strategy registry and the weight decay system forces re-evaluation over time. You can review and roll back any action from the dashboard.
Can it watch non-Docker services? +
Yes. The daemon monitors both Docker events and systemd/syslog. Any service visible in journald is auto-discovered and added to the watch list.
How is this different from PagerDuty or Datadog? +
Those tools alert you and wait for a human. SRE Daemon acts — it tries known fixes first (zero cost), escalates to AI only when needed, and learns from every outcome. You get paged only when all automated options are exhausted.
What's the roadmap after Linux? +
Kubernetes pod monitoring in Q3 2026, HTTP/API uptime checks in Q4 2026, and a security anomaly detection module planned for 2027 as a separate product line.
